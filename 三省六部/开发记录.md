## 📜 设计背景

### 为什么采用三省六部制？

传统 AI 助手通常是"单兵作战"模式，存在以下问题：
- 任务复杂时容易混乱
- 缺乏质量审核机制
- 没有明确的责任分工
- 难以追溯和复盘

唐朝三省六部制提供了一个成熟的**分权制衡 + 专业分工**架构：
- **三省**：决策（中书）→ 审核（门下）→ 执行（尚书），形成闭环
- **六部**：按职能专业化分工，按需调用

   │ (决策)  │    │ (审核)  │    │ (执行)  │

### 三省职责

| 省 | 职责 | 现代映射 | 关键能力 |
|---|------|---------|---------|
| **中书省** | 决策/起草诏令 | 任务规划/指令生成 | 任务分析、路径规划、资源分配 |
| **门下省** | 审议/审核 | 质量审核/风险控制 | 准确性检查、安全评估、风险识别 |
| **尚书省** | 执行 | 任务执行/协调 | 进度跟踪、部门协调、结果汇总 |

### 六部职责

| 部 | 职责 | 现代映射 | 关键能力 |
|---|------|---------|---------|
| **吏部** | 人事/官员管理 | 用户管理/记忆查询 | 用户画像、历史检索、偏好分析 |
| **户部** | 财政/户籍 | 资源管理/成本核算 | 预算规划、成本分析、资源调度 |
| **礼部** | 礼仪/教育/外交 | 对外沟通/文档 | 文档撰写、内容润色、多语言 |
| **兵部** | 军事 | 安全防御 | 安全审计、风险评估、应急预案 |
| **刑部** | 司法/刑狱 | 法务合规/错误处理 | 合规审查、错误分析、日志审计 |
| **工部** | 工程/建设 | 开发/运维 | 代码开发、系统部署、自动化 |

## 📋 工作流程

### 标准任务流程（三省制）

```
用户任务
   │
   ▼
┌─────────────┐
│ 中书省规划  │ → 任务分析 → 执行计划 → 分配方案
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 门下省审核  │ → 质量检查 → 风险评估 → 通过/修改
└──────┬──────┘
       │ (审核通过)
       ▼
┌─────────────┐
│ 尚书省执行  │ → 协调六部 → 进度跟踪 → 结果汇总
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 门下省汇报  │ → 任务总结 → 质量评估 → 归档记录
└─────────────┘
```

### 简单任务流程（直调六部）

```
用户任务
   │
   ▼
┌─────────────┐
│ 直接调用部  │ → 礼部写文档 / 工部写代码 / 吏部查记忆
└──────┬──────┘
       │
       ▼
   返回结果
```

## 💬 沟通协议

### 用户 ↔ 三省

**大黄主要与三省沟通：**
- 任务下达 → 中书省
- 进度查询 → 尚书省
- 结果验收 → 门下省

### 三省 ↔ 六部

**任务分配：**
- 中书省 → 尚书省 → 六部
- 紧急任务可直接调用六部

### 汇报机制

**任务完成时：**
1. 尚书省汇总执行结果
2. 门下省进行质量审核
3. 门下省向大黄汇报总结
